# writer-article — note記事 執筆スキル

## いつ使うか
ユーザーが「note記事を書いて」「このネタで記事にして」「直近のAIニュースをまとめ記事にして」と言ったとき。

## 前提
- 開始前に CLAUDE.md の「人格・ブランド定義」「品質基準」「サービス定義」を読む。
- `data/atoms.csv` を読み、対象atomを特定する（指定がなければ status=new のものから関連する複数件をまとめる）。
- `reference/` に過去記事アーカイブがあれば検索し、同テーマの焼き直しを避ける。

## 読者像
- AIにそれほど詳しくない一般企業の社員・担当者を想定する。
- 専門用語・略語・製品名は、初出で必ず1行のやさしい補足を入れる（例：「Codex（コードを書くAIツール）」）。
- 「で、自分の会社には何の関係があるのか」が必ず分かるように書く。

## 構成テンプレート（この順で書く）
1. タイトル: 内容を正確に表す。煽らない。30字前後
2. リード（導入3〜5行）: 何が起きたか／なぜ実務に効くか。専門知識がなくても読み進められる平易さにする
3. 本文: **ニュース（atom）ごとにH2見出しを立て、下記6ブロックを必ずこの順で書く**
4. まとめ: 3行で全体の要点を再掲
5. 締め: 読者への問いかけ1行（売り込みはしない）

### 本文の各ニュースで必ず書く6ブロック（見出しレベルを厳守）
ニュース1件につき、以下の構造で書く。**タイトルはH2（`##`）、それ以外の5ブロックはH3（`###`）の見出し**を立てて順番に書く。見出しレベルを揃えることで、初心者が構造を追いやすくする。

```markdown
## ① ニュースのタイトル（一言で表す見出し）

### 要点
まず結論。1〜2文、または箇条書き2〜3点で「何が起きたか」を最短で伝える

### 内容
何が発表され、何が変わるのかを、前提知識のない人にも分かるよう具体的に説明する。専門用語は補足する

### 深い考察
なぜこれが重要か、業界的に何を意味するか。推測は「〜と考えられる」、確定情報は言い切りで区別する

### 一般企業で考えるべきこと
自社（AIの専門企業でない普通の会社）が、これを受けて何を検討・準備すべきかを具体的なアクションで示す

### 引用URL
そのニュースの出典（atomのurl）を明記する
```

- タイトル（H2）にはニュースの通し番号（①②…）を付けてよい。
- 5つのH3見出しの文言（要点 / 内容 / 深い考察 / 一般企業で考えるべきこと / 引用URL）はそのまま使う。太字での代用はしない。

## トーンルール
- CLAUDE.md準拠。誇張語禁止。専門用語は初出で1行補足。
- 各事実には必ず出典（atomのurl）を、各ニュースの「引用URL」ブロックに置く。
- 推測は「〜と考えられる」、確定情報は言い切り、と区別する。
- 初心者向けを徹底する。難しい話を避けるのではなく、難しい話をやさしく言い換える。

## 手順
1. 対象atomを選び、内容と出典URLを把握する
2. 過去記事（reference/）と矛盾しないか確認する
3. 上記テンプレで本文を執筆する
4. `drafts/YYYY-MM-DD-note-記事スラッグ.md` に出力する
5. 使ったatomの status を `picked` に更新する
6. `pipeline.csv` に行を追加（channel=note, status=drafted, planned_date=今日）
7. 完了をユーザーに報告し、次は review スキルでの事実確認を促す

## 品質チェック（Output Gate）
- [ ] 各ニュースが「タイトル→要点→内容→深い考察→一般企業で考えるべきこと→引用URL」の6ブロックを順に満たしている
- [ ] タイトルがH2（`##`）、残り5ブロックがH3（`###`）で構造化されている
- [ ] 専門用語・製品名が初出でやさしく補足されている（初心者が読める）
- [ ] すべての数字・機能名・日付に出典URLがある
- [ ] 元ソースが言っていないことを断定で書いていない
- [ ] 誇張語を使っていない
- [ ] 過去記事と重複していない

## 出力先
- 記事本文: `drafts/`（Markdown）
- 状態更新: `data/atoms.csv`, `data/pipeline.csv`
