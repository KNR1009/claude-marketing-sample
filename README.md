# AIニュース → note記事 / X投稿 自動化パイプライン

株式会社サードスコープのマーケティング用リポジトリです。
**「AIニュースを集める → note記事を書く → X投稿案を作る → 事実確認する」** という一連の流れを、Claude Code の **スキル（Skill）** で半自動化します。

このREADMEは、**作成者以外でも運用できる**ことを目的に、全体像・各スキルの責務・実行方法・確認方法をまとめています。

---

## 1. 全体像（30秒で理解する）

```
        ┌─────────────────────────────────────────────────────┐
        │                  daily-pipeline                     │
        │              （司令塔／順番に回すだけ）              │
        └─────────────────────────────────────────────────────┘
                 │           │            │            │
                 ▼           ▼            ▼            ▼
           ┌──────────┐ ┌───────────┐ ┌──────────┐ ┌────────┐
           │research- │ │writer-    │ │writer-   │ │review  │
           │ai        │→│article    │→│post      │→│        │
           │収集      │ │note記事   │ │X投稿     │ │品質判定│
           └──────────┘ └───────────┘ └──────────┘ └────────┘
                 │           │            │            │
                 ▼           ▼            ▼            ▼
           data/atoms   drafts/note   drafts/x-    PASS/FIX/
           .csv に追記   .md を生成    posts.md     BLOCK判定
```

- **データ（ネタの種）は `data/` のCSV** に貯まり、**成果物（下書き）は `drafts/`** に出力されます。
- **外部への実投稿・公開はしません。** 下書きを作るところまでが役割です（公開は人が手動で行う）。

## 3. 各スキルの責務と動き

| スキル | レイヤー | 役割（何をするか） | 入力 | 出力 |
|---|---|---|---|---|
| **research-ai** | 調査 | AIニュースを集め、一次情報で裏取りして種を貯める | 期間・対象ベンダー | `data/atoms.csv` に追記 |
| **writer-article** | 制作 | 種から note記事の下書きを書く | `atoms.csv` | `drafts/*-note-*.md` |
| **writer-post** | 制作 | 種 or 記事から X投稿案（A/B）を作る | `atoms.csv` / note記事 | `drafts/*-x-posts.md` |
| **review** | 品質管理 | 事実確認し、公開可否を判定する | `drafts/` + `atoms.csv` | 判定（PASS/FIX/BLOCK） |
| **daily-pipeline** | 司令塔 | 上記4つを順番に回す（ロジックは持たない） | — | 全成果物を一括生成 |

### それぞれの中身（要点）

- **research-ai**
  - 対象: Anthropic(Claude) / Google(Gemini) / OpenAI。期間は既定で「今日から3日間」。
  - **公式ブログ等の一次情報URLを必ず確保**。二次情報のみは要注意マークを付ける。
  - 既存 `atoms.csv` と重複（同URL/同タイトル）は追加しない。

- **writer-article**
  - 読者像は「**AIに詳しくない一般企業の社員**」。専門用語は初出でやさしく補足。
  - 各ニュースを **H2タイトル ＋ H3で「要点／内容／深い考察／一般企業で考えるべきこと／引用URL」** の6ブロック構成で書く。

- **writer-post**
  - 全角140字以内。1ネタにつき **A案（事実訴求）/ B案（問いかけ訴求）** の2案。
  - ハッシュタグ最大2個。**実投稿はしない**（下書きのみ）。

- **review**（＝Output Gate）
  - 5観点で判定: **事実 / 出典 / 帰属（ベンダー取り違え） / 誇張 / 重複**。
  - 総合判定: **PASS（公開可）/ FIX（要修正）/ BLOCK（出典不足で不可）**。
  - PASS時のみ `pipeline.csv` を `reviewed` に更新。公開操作はしない。

- **daily-pipeline**（司令塔）
  - **自分でロジックを持たず、各スキルの手順をそのまま呼ぶ**だけ。
  - 重複ガード・各STEPのチェック・最大2回までの自動修正ループを管理する。

## 4. 実行方法

Claude Code のチャットで、**スキル名 or 起動フレーズ**を入力するだけです。

### A. まとめて全部回す（推奨）

```
daily-pipeline を実行して
```
または「今日のパイプライン回して」「一連の流れを自動でやって」。
→ 収集→記事→X投稿→レビュー まで一気に走り、`drafts/` に成果物が揃います。

> 開始時に「期間（既定3日）」「v2を別ファイルにするか」等を聞かれることがあります。指定がなければ既定値で進みます。

### B. 1ステップずつ回す

| やりたいこと | 入力例 |
|---|---|
| ニュース収集だけ | 「直近3日のClaude/Gemini/OpenAIのアップデートを research-ai で集めて」 |
| 記事だけ書く | 「さっきのatomsで writer-article でnote記事を書いて」 |
| X投稿だけ作る | 「この記事の告知ポストを writer-post で作って」 |
| レビューだけ | 「drafts の最新をレビューして」 |

### C. パラメータ（daily-pipeline 既定値）

- 対象ベンダー: anthropic / google / openai
- 期間: 今日から3日間
- 記事の想定文字数: 2000〜3000字
- 記事本数: 1本（複数ニュースをまとめる）

## 5. 確認方法（運用者がチェックすべきポイント）

### 成果物を見る
- 記事 / 投稿の下書き → **`drafts/` フォルダ**の `.md` ファイル。
- 進捗状況 → **`data/pipeline.csv`** の `status` 列。

### `pipeline.csv` の status の意味

| status | 状態 |
|---|---|
| `drafted` | 下書き生成済み（未レビュー） |
| `reviewed` | レビューPASS済み（**公開してよい状態**） |

### 公開前チェックリスト（人がやること）
- [ ] `drafts/` の本文に目を通し、トーン・内容に違和感がないか
- [ ] **出典URLが実際に開けるか**（特にreviewで「要目視」と言われたもの）
- [ ] 数字・固有名詞・日付が正しいか（reviewの判定がPASSでも最終責任は人）
- [ ] X投稿が140字以内に収まっているか

### 公開したら（記録を残す）
公開後、`data/outputs.csv` に1行追記します（書式はreview実行後に提示されます）。
```csv
id,channel,title,url,published_date,atoms
o1,note,記事タイトル,公開後のURL,2026-06-13,a1;a2;a3
```
→ これにより、次回以降 review が**焼き直し（重複）を検知**できるようになります。

## 6. データファイルの読み方

### `data/atoms.csv`（ニュースの種）
| 列 | 意味 |
|---|---|
| id | 連番（a1, a2…） |
| date | 発表日 |
| source | 取得元（◯◯公式 など） |
| vendor | anthropic / google / openai |
| title | 見出し |
| url | 一次情報URL |
| summary | 80〜120字の要約 |
| status | `new`（未使用）→ `picked`（記事に使用済み） |

### `data/pipeline.csv`（制作中の進捗）
`id, channel(note/x), title, draft_path, atoms, status, planned_date`

### `data/outputs.csv`（公開済み記録）
`id, channel, title, url, published_date, atoms`（公開後に人が追記）

## 7. 大事なルール（CLAUDE.md より）

このリポジトリの全スキルは `CLAUDE.md` を**共通の憲法**として守ります。

- **品質**: 数字・固有名詞は出典を確認してから書く。断定できないことは「〜とされる」と書く。
- **トーン**: 専門的だが平易。誇張表現（神／最強 等）は使わない。
- **承認ルール**: **ファイル生成は自動でOK。外部投稿・送信（X / メール等）は必ず人が確認**。
  - → このパイプラインは**下書きまで**。実際の投稿・公開は必ず人の手で行ってください。

## 8. よくあるつまずき（FAQ）

**Q. 収集したのに記事に出てこないニュースがある**
A. 期間外、または一次情報で日付が確定できなかったものは、品質基準に従って除外されます。理由は実行時のチャットに報告されます。

**Q. Geminiのニュースが0件になることがある**
A. その期間にGoogleの公式発表がなければ0件です。誤って古い発表（例: I/Oの内容）を混ぜないよう除外しています。

**Q. reviewで「URLが403」と言われた**
A. サイト側のbot対策で自動チェックが弾かれているだけのことがあります。その場合は**ブラウザで開いて目視確認**してください。

**Q. 同じ日にもう一度パイプラインを回したい**
A. 重複ガードが働き「本日分は既にあります。再生成しますか？」と確認されます。v2として別ファイルに残すことも可能です。

## 9. 標準的な1日の運用フロー

1. チャットで `daily-pipeline を実行して` と入力
2. 期間など聞かれたら答える（既定でよければそのまま）
3. 生成された `drafts/` の記事・投稿に目を通す
4. reviewの判定が **PASS** で、出典URLも問題なければ公開準備OK
5. note / X に**手動で**公開
6. `data/outputs.csv` に公開記録を1行追記
